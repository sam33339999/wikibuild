# Spec: v1.1 — SEO 欄位 · LLM 摘要 · 寫作搜尋 · MCP

Status: **planned** (not implemented)  
Depends on: v1.0 shipped (M0–M7)  
Owner track: product next after Claude UI / floating TOC

## Goal

讓作者能**明確控制**分享與搜尋用的中繼資料；用 **LLM** 加速摘要／索引；寫作時有 **站內 AI 輔助搜尋**；並用 **MCP** 讓外部代理讀寫文章。

## Non-goals (this track)

- 多管理者／RBAC
- 版本歷史、備份還原
- ActivityPub / 完整 public REST API 產品化（MCP 可先覆蓋自動化需求）
- 自動產生 OG 圖（可之後接）

## Dependency order

```
S1 SEO / media fields  ──►  S2 LLM generate into those fields
                         └►  S3 AI search while writing (uses body + index)
S4 MCP article CRUD    ──►  (prefer after S1 so create/update carry SEO)
```

Ship S1 first. S2 and S4 can parallel after S1 schema is stable. S3 needs search quality + editor hooks.

## S1 — Editable SEO & social fields

### Problem

Today SEO is **automatic only** (title + `feedSummary(body)` ≤160 bytes, optional `WIKIBUILD_BASE_URL`). Authors cannot set description, OG image, or SEO title ≠ display title. No admin UI for SEO.

### Data model (Article)

New nullable / empty-string columns (names tentative; sqlc + migration):

| Field | Type | Use |
|-------|------|-----|
| `seo_title` | text, default `''` | `<title>`, `og:title`, `twitter:title` if set; else `title` |
| `summary` | text, default `''` | Human/AI blurb (lists, feeds optional); distinct from meta desc if both set |
| `meta_description` | text, default `''` | `meta description`, `og:description`, JSON-LD description |
| `cover_image_url` | text, default `''` | Optional on-page hero / list card |
| `og_image_url` | text, default `''` | OG/Twitter image only; **not required = cover** |

**Fallback chain (public render):**

1. `meta_description` → else `summary` → else auto `feedSummary(body)` (improve to rune-safe ~155 chars, strip more MD noise).
2. `seo_title` → else `title`.
3. `og_image_url` → else `cover_image_url` → else omit image / `summary` card.
4. Canonical / JSON-LD URL still need `WIKIBUILD_BASE_URL`.

### Admin UI

- Markdown + HTML-upload edit forms: section **「SEO / 分享」**
  - SEO 標題、摘要、Meta description、封面 URL、OG 圖 URL
  - Short help: OG 圖 ≠ 封面；空則自動
- Optional: pick from already-uploaded media (`/media/...`)
- List page: no need to show all SEO columns; optional badge “缺 description”

### Public / feeds / sitemap

- `layout.SEO` filled from article fields + fallbacks
- JSON-LD `BlogPosting`: headline = effective title; description = effective meta; `image` if OG/cover
- RSS/Atom/JSON Feed item summary prefer `summary` then meta then body clip
- Sitemap: no change required (slugs only); later `lastmod` from `updated_at` if not already

### Acceptance

- [ ] Migration + model + inmem + postgres + form bind (`strings.Clone`)
- [ ] View-source on published article shows author-set meta when set
- [ ] Empty fields keep current auto behaviour
- [ ] Unit tests: fallback matrix; handler form round-trip
- [ ] `make test` green

### Estimate

M-sized (schema + admin + public wiring).

## S2 — LLM: 索引 → meta description → 摘要

### Problem

Filling S1 fields by hand is slow. Authors want one action: read body → produce outline/index, meta description, summary.

### Behaviour

- Admin article edit: button **「AI 產生 SEO」** (markdown primarily; html_upload optional later)
- Pipeline (single request or chained prompts):
  1. **Index / outline** — bullet structure or keyword list (store? v1: return to UI only, or optional `ai_index` text field later)
  2. **Meta description** — ~120–155 chars, plain text, no markdown
  3. **Summary** — 2–4 sentences for humans / feeds
- Results **pre-fill form fields**; author must **Save** to persist (no silent overwrite of body)
- Idempotent regenerate; confirm if fields non-empty

### Config

| Env / setting | Purpose |
|---------------|---------|
| LLM API key + base URL | Prefer configurable provider (see build-with-ai / SpaceXAI preference when scaffolding) |
| Model name | e.g. fast chat model |
| Feature flag | empty key → button disabled with message |

No keys in git. Optional settings row later; env-first is OK for v1.

### Architecture

- `internal/llm/` — client interface + one impl (HTTP)
- Handler `POST /admin/:id/ai/seo` (auth + CSRF) or generate without id for new drafts (body in POST)
- Pure prompt builders unit-tested with golden snippets
- Timeouts, max body size, rate limit per admin session

### Acceptance

- [ ] With key set: button fills meta_description + summary (and shows outline)
- [ ] Without key: clear error, no panic
- [ ] Does not publish or change status
- [ ] Unit tests for prompt construction + response parse; handler mock client

### Estimate

M–L (provider + UI + safety).

## S3 — Writing-time AI-assisted search

### Problem

While drafting, authors want related internal posts / wikilink targets without leaving the editor.

### Behaviour (phased)

**S3a — site search in editor (no LLM required)**  
- Editor toolbar or side panel: query → search published+draft admin articles (existing search)
- Insert `[[slug]]` or markdown link

**S3b — AI-assisted**  
- Given selection or current paragraph: LLM suggests related slugs + short reason from site index (titles, tags, summaries from S1)
- User clicks to insert wikilink

### Acceptance

- [ ] S3a: search + insert wikilink from admin editor
- [ ] S3b: optional LLM suggestions; works offline path without key (S3a only)
- [ ] Does not leak private articles to public APIs

### Estimate

S3a S; S3b M after S1–S2.

## S4 — MCP server (articles)

### Problem

External agents (Cursor, Claude, etc.) need a stable tool surface to list/get/create/update articles and change status without scraping HTML admin.

### Surface (tools)

| Tool | Action |
|------|--------|
| `list_articles` | filter status/visibility/q; pagination |
| `get_article` | by id or slug |
| `create_article` | markdown draft default; fields incl. S1 SEO when present |
| `update_article` | patch title/body/tags/SEO… |
| `set_article_status` | draft \| published (+ optional publish_at) |
| `set_article_visibility` | public \| protected \| private |

### Auth & safety

- MCP auth token (env `WIKIBUILD_MCP_TOKEN` or similar); reject empty in production
- Prefer bind localhost or reverse-proxy only
- Create defaults: `status=draft`, `visibility=private` or `public` (decide: **draft + private** safest)
- No password hashes in responses
- Audit log optional later

### Shape

- Subcommand or package: `cmd/wikibuild-mcp` **or** same binary `wikibuild mcp`
- Reuse `store.Repository` + auth; no duplicate business rules
- Transport: stdio MCP for local agents first

### Acceptance

- [ ] Documented tool schemas + example agent config
- [ ] Create draft → get → publish status via tools
- [ ] Invalid token → fail closed
- [ ] Tests against inmem store

### Estimate

M (stdio MCP + auth + tools).
